# Codex Compatibility Surface Projection Phase 3

Generated: 2026-03-26

## Objective

Complete the remaining Codex-authored compatibility surfaces in the non-destructive `definitions/ -> projected surface` architecture so `.codex/` keeps working as a runtime surface while authored wrappers, docs, and helper templates originate from one stable source tree.

## Problem Statement

The repository already projects Codex skills and orchestration from `definitions/`, but a small set of Codex compatibility assets still lives directly inside `.codex/`:
- `.codex/mcp/README.md`
- `.codex/mcp/codex.config.template.toml`
- `.codex/mcp/vscode.mcp.template.json`
- `.codex/scripts/README.md`
- `.codex/scripts/render-vscode-mcp.ps1`
- `.codex/scripts/sync-mcp-to-codex-config.ps1`

That leaves authority split between `definitions/` and projected runtime folders. The remaining safe slice is to move these authored support assets behind `definitions/providers/codex/` while leaving the generated `servers.manifest.json` driven by the canonical MCP runtime catalog.

## Design Summary

This phase stays additive and non-destructive:
- `definitions/providers/codex/mcp/` owns authored MCP support docs/templates
- `definitions/providers/codex/scripts/` owns authored compatibility wrappers/docs
- `.codex/mcp/servers.manifest.json` remains generated from `.github/governance/mcp-runtime.catalog.json`
- `.codex/mcp/` and `.codex/scripts/` remain tracked projected/runtime surfaces
- `scripts/runtime/render-codex-compatibility-surfaces.ps1` becomes the renderer for the remaining authored compatibility assets

### Authoritative additions in this phase

- `definitions/providers/codex/mcp/README.md`
- `definitions/providers/codex/mcp/codex.config.template.toml`
- `definitions/providers/codex/mcp/vscode.mcp.template.json`
- `definitions/providers/codex/scripts/README.md`
- `definitions/providers/codex/scripts/render-vscode-mcp.ps1`
- `definitions/providers/codex/scripts/sync-mcp-to-codex-config.ps1`

### Projected surfaces affected

- `.codex/mcp/README.md`
- `.codex/mcp/codex.config.template.toml`
- `.codex/mcp/vscode.mcp.template.json`
- `.codex/scripts/**`

## Decisions

1. `servers.manifest.json` remains generated by the canonical MCP catalog renderer and is not moved into `definitions/providers/codex/mcp/`.
2. `.codex/scripts/` stays as a compatibility surface for external/runtime expectations, but its authored wrapper content is rendered from `definitions/providers/codex/scripts/`.
3. Bootstrap must render the new Codex compatibility surfaces before mirroring `.codex/` into the shared runtime.
4. Validation must separately check generated-manifest parity and authored-compatibility parity.
5. The architecture continues to reserve `scripts/` as the only operational layer; wrapper scripts in `.codex/scripts/` remain thin forwarders only.

## Constraints

- Do not introduce Rust/Cargo in this phase.
- Do not remove `.codex/` or convert runtime consumers to new paths.
- Do not merge generated MCP manifest ownership into authored compatibility files.
- Keep install/bootstrap behavior stable for operators.

## Acceptance Criteria

1. The authored files under `.codex/mcp/` except `servers.manifest.json` are sourced from `definitions/providers/codex/mcp/`.
2. `.codex/scripts/` is sourced from `definitions/providers/codex/scripts/`.
3. `scripts/runtime/render-codex-compatibility-surfaces.ps1` renders both compatibility surfaces deterministically.
4. `scripts/runtime/bootstrap.ps1` invokes the Codex compatibility renderer before Codex runtime sync.
5. `scripts/validation/validate-instructions.ps1` detects drift between the authoritative Codex compatibility definitions and the projected surfaces.
6. `scripts/tests/runtime/runtime-scripts.tests.ps1` includes parameter and smoke coverage for the new renderer.
7. `validate-all` and the full `install.ps1 -RuntimeProfile all ...` flow remain green.

## Risks

- Over-mirroring `.codex/mcp/` could delete the generated manifest.
- Wrapper README/docs may drift if rendered copies are not updated together.
- Bootstrap may start syncing stale files if the new renderer is not invoked consistently.

## Mitigations

- Use selected-file rendering for `.codex/mcp/` support files and keep the manifest under the existing catalog renderer.
- Mirror the full `.codex/scripts/` surface from `definitions/providers/codex/scripts/`.
- Cover both parity and renderer smoke tests before install validation.
